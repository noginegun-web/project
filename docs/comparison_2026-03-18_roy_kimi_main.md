## Краткий вывод

Лучшая база для продолжения работы - текущий основной проект.

Причина простая: только он уже прошёл через реальные локальные и hosted-прогоны, а значит даёт не теоретические предположения, а проверенные причины поломок. У Kimi сильный аудит и хороший список разрывов с Oxygen. У Roy полезные замечания по упаковке и сборке плагинов, но его архив не является альтернативным полноценным фреймворком.

## Итог по качеству

### 1. Main (наш текущий проект)

Сильные стороны:
- есть живой `version.dll -> hostfxr -> .NET runtime -> native bridge -> web`
- есть реальные hosted-логи и локальные логи
- уже выявлен главный production-блокер: зависание в native hook path на хостинге
- уже проверены реальные симптомы:
  - кик игрока после `/hello`
  - `Hang detected on GameThread`
  - зависание внутри `ScumOxygen.Native.dll`
- уже поднят веб, API, hot-reload, plugin web routes, diagnostics
- уже собран практический опыт по Ark Hoster и реальной выкладке

Слабые стороны:
- hook-layer пока нестабилен в hosted-режиме
- часть parity с Oxygen до сих пор только частичная
- в сборке до этого сохранялся лишний мусорный legacy-слой `Plugins`

Итог:
- это лучшая рабочая база
- именно её надо развивать дальше

### 2. Kimi (OpenClaw параллельный аудит)

Сильные стороны:
- лучший стратегический аудит разрывов с Oxygen
- хорошо выделяет архитектурные gaps:
  - надёжность команд
  - ACK/retry в native bridge
  - Web API gaps
  - HTTP client gaps
  - inventory/equipment parity gaps
- полезен как второй независимый взгляд
- даёт понятный roadmap и таблицу приоритетов

Слабые стороны:
- местами слишком оптимистично оценивает уже реализованные хуки
- не опирается достаточно жёстко на реальные hosted-краши
- часть выводов устаревает, если смотреть только документы и код без постоянной прогонки на хостинге

Итог:
- лучший как аудит и планировщик
- плох как единственный источник истины

### 3. Roy

Сильные стороны:
- правильно заметил проблему с упаковкой sample plugin
- правильно обратил внимание на конфликтную доставку `ScumOxygen.Core.dll`
- верно указал, что рабочая директория плагинов должна быть `oxygen/plugins`

Слабые стороны:
- это не полноценный фреймворк, а маленький пример плагина плюс заметки
- часть утверждений уже устарела относительно текущего runtime и hosted-фактов
- он не нашёл главный production-блокер с живым игроком
- не даёт полноценного решения по native hooks, direct memory access и hosted stability

Итог:
- полезен точечно
- не может быть основной технической базой проекта

## Что у Roy реально полезно и уже стоит учитывать

1. Не тащить в plugin output лишний `ScumOxygen.Core.dll`.
2. Не плодить устаревшую директорию `Plugins`, если runtime реально работает через `oxygen/plugins`.
3. Не смешивать sample precompiled plugin с основным runtime-путём без необходимости.

## Что у Kimi реально полезно и уже стоит учитывать

1. Доделать надёжность native bridge:
   - ACK
   - retry
   - better command delivery tracing
2. Доделать parity по HTTP client:
   - `POST`
   - `PUT`
   - `DELETE`
   - headers
3. Доделать process-first inventory/equipment/map.
4. Считать DB и log-пути только fallback-слоем, а не основным engine-path.

## Что уже доказано только нашей основной веткой

1. Проблема кика игрока на хостинге воспроизводится именно в реальном живом входе.
2. Основной production-блокер сейчас сидит в native/hook части, а не в простом вебе или в sample plugin.
3. Простое удаление `anti-vpn` не решает проблему.
4. Простая очередь на pipe тоже не решает проблему до конца.
5. Нынешний `ProcessEvent` path требует переделки или безопасного временного обхода.

## Текущий честный рейтинг

1. Main - лучшая реальная база и главный источник истины
2. Kimi - лучший аудит и список следующих шагов
3. Roy - полезные упаковочные правки, но не альтернатива движку

## Что уже сделано после сравнения

На основе замечаний Roy уже внесены полезные cleanup-правки:

1. В `ScumOxygen.SamplePlugin.csproj` отключено копирование зависимости `ScumOxygen.Core` в output плагина.
2. В `Build-HostedPackage.ps1` убрана упаковка legacy-слоя:
   - `NeDjin\\Plugins`
   - `ScumOxygen\\Plugins`
   - precompiled sample plugin DLL
   - лишний `ScumOxygen.Core.dll` рядом с sample plugin

Это не решает главный hosted-hook crash, но уменьшает шум, ложные причины и конфликтные сборки.

2. Доработать native hook layer уже на свежем dedicated dump.
3. После стабилизации command path возвращаться к full Oxygen parity:
   - equipment
   - inventory
